# Debt Collection Decision Pipeline

Проект строит единый data/ML-пайплайн для управления взысканием задолженности по лицевым счетам (ЛС). Он принимает исходные Excel/CSV-выгрузки из папки `ТЗ`, приводит их к нормальной аналитической структуре, строит месячную витрину признаков, сегментирует абонентов, оценивает эффект мер воздействия и готовит результаты для анализа и презентации.

Главная точка входа:

```bash
python main.py
```

Команда запускает локальный сайт на `localhost`. В интерфейсе можно увидеть все этапы пайплайна, запустить полный цикл обработки, следить за прогрессом и открыть готовые таблицы, графики и markdown-материалы.

## Зачем нужен этот пайплайн

В задаче взыскания важно не просто найти должников, а выбрать корректное действие для каждого ЛС: напоминание, звонок, уведомление, ограничение, претензию, юридический маршрут или отсутствие действия. Для этого нужны три вещи:

1. Чистые и сопоставимые данные по ЛС, оплатам, задолженности и мерам воздействия.
2. Понятная сегментация клиентов по риску, поведению и бизнес-ограничениям.
3. Оценка ожидаемого эффекта меры, чтобы не применять действие только потому, что оно часто использовалось в прошлом.

Пайплайн решает эти задачи последовательно: сначала делает данные надежными, затем формирует признаки, затем объединяет бизнес-правила и ML, а в конце строит рекомендации и материалы для объяснения результата.

## Структура проекта

```text
case/
  main.py                         # единая точка входа: запускает локальный сайт
  pipeline/
    settings.py                   # централизованные настройки
    web_app.py                    # localhost-интерфейс
    run_pipeline.py               # единый оркестратор этапов
    prepare_raw.py                # raw-слой
    build_normalized.py           # нормализация и DQ
    build_mart.py                 # месячная витрина признаков
    build_segmentation.py         # rule + KMeans + hybrid gates
    build_uplift_prototype.py     # uplift-прототип и рекомендации
    build_presentation_viz.py     # графики
    build_presentation_notes.py   # структура доклада и заметки
  ТЗ/                             # входные файлы 01..14
  output/                         # результаты пайплайна
```

Параметры командной строки не используются. Основные настройки лежат в `pipeline/settings.py`:

```python
SOURCE_DIR = ROOT_DIR / "ТЗ"
OUTPUT_DIR = ROOT_DIR / "output"
HOST = "127.0.0.1"
PORT = 8000
```

## Общая схема пайплайна

```text
ТЗ/*.xlsx, ТЗ/*.csv
  -> output/raw
  -> output/normalized
  -> output/mart
  -> output/segmentation
  -> output/uplift
  -> output/presentation
```

Единый оркестратор `pipeline/run_pipeline.py` запускает этапы строго по порядку:

1. `prepare_raw`
2. `build_normalized`
3. `build_mart`
4. `build_segmentation`
5. `build_uplift_prototype`
6. `build_presentation_viz`
7. `build_presentation_notes`

Такой порядок выбран потому, что каждый слой зависит от предыдущего: нельзя корректно считать uplift, пока нет витрины признаков; нельзя строить витрину, пока данные не нормализованы; нельзя объяснять результат, пока нет сегментов, рекомендаций и графиков.

## Этап 1. Raw-слой: `prepare_raw`

### Что происходит

Каждый входной файл переводится в единый CSV-формат:

```text
source_file, source_row_num, col_1, col_2, col_3, ...
```

Для Excel используется чтение первой вкладки напрямую из XML-структуры `.xlsx`. Для CSV используется разделитель `;` и кодировка `utf-8-sig`.

На выходе создаются файлы:

```text
output/raw/01_ls_profile_raw.csv
output/raw/02_balance_raw.csv
output/raw/03_payment_raw.csv
...
output/raw/14_action_limits_raw.csv
output/raw/load_metrics_raw.json
```

Если установлен `pyarrow`, дополнительно пробуется запись Parquet. CSV создается всегда.

### Почему выбран такой подход

Raw-слой сохраняет исходную структуру почти без интерпретации. Это важно для трассируемости: если на позднем этапе появилась странная запись, ее можно найти по `source_file` и `source_row_num`.

Пример: если в нормализации найден нечисловой ЛС `#N/A`, в DQ-таблице будет видно, из какого исходного файла и строки он пришел.

## Этап 2. Нормализация и DQ: `build_normalized`

Нормализация превращает “колонки по позициям” из raw-слоя в канонические таблицы с понятными полями.

### Общие преобразования

| Тип значения | Как обрабатывается | Зачем |
|---|---|---|
| ЛС | Оставляются только строки цифр, значение вида `123.0` превращается в `123` | Единый ключ для всех источников |
| Даты | Поддерживаются ISO, `dd.mm.yyyy` и Excel serial date | В источниках даты могут быть записаны по-разному |
| Суммы | Пробелы убираются, запятая заменяется на точку, используется `Decimal` | Корректная работа с рублями и копейками |
| Да/Нет | `Да`, `Нет`, `1`, `0`, `true`, `false` приводятся к `1/0` | Бинарные признаки нужны для витрины и моделей |
| GUID | Проверяется валидность UUID | Адрес должен быть технически корректным |
| Ошибки | Не ломают пайплайн, а записываются в `dq_issues.csv` | Модель продолжает работать, но качество данных прозрачно |

### `dim_ls_profile`

Из файла `01` строится профиль ЛС:

```text
ls_id
can_remote_disconnect
has_phone
has_benefits
is_gasified
is_city
has_yaroblierc_receipt
has_post_receipt
has_e_receipt
not_resident
is_chd
is_mkd
is_dormitory
meter_install_tambour
meter_install_pole
meter_install_apartment_house
meter_install_stairwell
address_guid
source_record_count
```

Если один ЛС встречается несколько раз, записи схлопываются в одну. Для бинарных признаков берется `max`: если хотя бы в одной строке указано `Да`, итоговый признак становится `1`.

Почему так: в профиле ЛС наличие признака обычно важнее его отсутствия. Например, если в одной строке телефон есть, а в другой пусто или `Нет`, безопаснее считать, что канал контакта потенциально доступен. Конфликт при этом не скрывается: он записывается в DQ.

Пример:

| ls_id | has_phone | meter_install_pole |
|---|---:|---:|
| 100 | 1 | 0 |
| 100 | 1 | 1 |

Итог:

| ls_id | has_phone | meter_install_pole | source_record_count |
|---|---:|---:|---:|
| 100 | 1 | 1 | 2 |

### `fact_balance_monthly`

Файл `02` содержит месяцы в широком формате: после ЛС идут повторяющиеся тройки колонок.

```text
ЛС, СЗ на начало, Начислено, Оплачено, СЗ на начало, Начислено, Оплачено, ...
```

Пайплайн преобразует его в длинный формат:

```text
ls_id, period_month, debt_start, charge_amt, paid_amt, source_row_num
```

Первый месяц берется из заголовка как Excel serial date, следующие месяцы рассчитываются по позиции тройки колонок. Это важно, потому что в широких бухгалтерских выгрузках заголовки могут повторяться или быть неполными, а позиция месяца стабильнее.

Пример:

| ls_id | debt Jan | charge Jan | paid Jan | debt Feb | charge Feb | paid Feb |
|---|---:|---:|---:|---:|---:|---:|
| 1 | 100 | 20 | 5 | 115 | 25 | 0 |

Итог:

| ls_id | period_month | debt_start | charge_amt | paid_amt |
|---|---|---:|---:|---:|
| 1 | 2025-01-01 | 100 | 20 | 5 |
| 1 | 2025-02-01 | 115 | 25 | 0 |

### `fact_payment_txn`

Файл `03` превращается в транзакционную таблицу:

```text
ls_id, payment_date, amount_rub, payment_method, is_reversal, source_row_num
```

Отрицательная сумма считается корректировкой и получает `is_reversal = 1`. Пустой способ оплаты заполняется как `NULL`, чтобы отличать “нет значения” от случайной потери колонки.

Почему так: транзакции нужны для признаков активности оплат и для целевой переменной `paid_any_next_month`. Отдельный флаг корректировки помогает не смешивать обычную оплату и сторно.

### `fact_action_event`

Файлы `04..13` объединяются в одну таблицу мер воздействия:

```text
ls_id, action_date, action_type, event_count
```

События агрегируются по ключу:

```text
ls_id + action_date + action_type
```

Почему так: для витрины важен факт и частота воздействия, а не техническая строка в исходном файле. Если по одному ЛС было 3 СМС в один день, в таблице будет одна запись `event_count = 3`.

### `dim_action_limit`

Файл `14` превращается в справочник лимитов:

```text
action_type, month_limit, is_unlimited, source_measure_name
```

Если текст содержит `нет ограничений`, ставится `is_unlimited = 1`. В остальных случаях из строки извлекаются цифры лимита.

Почему так: лимиты мер нужны для будущего портфельного оптимизатора и для бизнес-интерпретации. Даже если текущий прототип не распределяет бюджет мер, справочник уже готов для следующего шага.

### `dq_issues`

Все проблемы качества данных складываются в отдельную таблицу:

```text
issue_id, stage, source_file, source_row_num, issue_type, issue_detail, ls_id_raw
```

Типовые DQ-события:

| issue_type | Когда возникает |
|---|---|
| `NON_NUMERIC_LS` | ЛС не является строкой цифр |
| `INVALID_YES_NO` | Бинарное поле нельзя привести к `Да/Нет` |
| `INVALID_UUID` | Адресный GUID невалиден |
| `INVALID_PAYMENT_DATE` | Дата оплаты не распознана |
| `INVALID_AMOUNT` | Сумма не распознана |
| `INVALID_ACTION_DATE` | Дата меры не распознана |
| `UNKNOWN_ACTION_TYPE` | Мера из лимитов не сопоставлена с типом |
| `INVALID_LIMIT_VALUE` | Лимит нельзя прочитать |

Почему DQ не останавливает процесс: в реальных выгрузках всегда есть шум. Остановить весь пайплайн из-за нескольких плохих строк хуже, чем изолировать эти строки и продолжить расчет на валидных данных.

## Этап 3. Месячная витрина признаков: `build_mart`

Витрина `mart_ls_month.csv` строит одну строку на пару:

```text
ЛС + месяц T
```

В нее попадают признаки, которые доступны на конец месяца `T`, и целевые переменные по следующему месяцу `T+1`.

### Основные группы признаков

| Группа | Примеры колонок | Что означает |
|---|---|---|
| Баланс | `debt_start_t`, `charge_amt_t`, `paid_amt_t`, `debt_change_1m` | Текущий долг, начисления, оплаты и изменение долга |
| Профиль | `has_phone`, `has_e_receipt`, `can_remote_disconnect`, `is_mkd` | Статичные признаки ЛС |
| Каналы контакта | `contact_channels_count` | Сумма доступных каналов: телефон, электронная, почтовая и ЯрОблИЕРЦ-квитанция |
| Оплаты | `payment_txn_count_l1m/l3m/l6m`, `payment_amount_l1m/l3m/l6m` | Количество и сумма оплат за окна 1/3/6 месяцев |
| Меры воздействия | `action_total_l1m/l3m/l6m`, `action_<type>_l1m/l3m/l6m` | Сколько мер применяли недавно |
| Цель | `paid_any_next_month`, `paid_amount_next_month`, `recovery_ratio_next_month` | Что произошло в следующем месяце |

Окна `l1m`, `l3m`, `l6m` включают месяц `T`.

Пример для февраля:

```text
payment_amount_l3m = оплаты за декабрь + январь + февраль
action_sms_l1m = количество СМС в феврале
paid_any_next_month = была ли оплата в марте
```

### Leakage-safe логика

Признаки используют только информацию до конца месяца `T`. Информация из `T+1` используется только как target для обучения.

Почему это важно: если модель увидит будущую оплату в признаках, она будет показывать красивое качество на истории, но провалится в реальном применении. Поэтому будущий месяц отделен как целевая переменная.

### Почему выбран месячный уровень

Меры взыскания и задолженность обычно анализируются в месячном управленческом цикле: лимиты мер, отчетность, начисления и планы по взысканию чаще всего месячные. Месячная гранулярность снижает шум дневных событий и остается достаточно детальной для принятия решений.

## Этап 4. Сегментация: `build_segmentation`

Сегментация состоит из трех уровней:

1. Rule-score: понятный экспертный риск-скоринг.
2. KMeans: ML-кластеры по поведению и финансовым признакам.
3. Hybrid gates: бизнес-гейты, которые решают, где ML допустим, а где должен работать только rule-based маршрут.

### Rule-score

Для каждого ЛС считается балл риска:

| Условие | Баллы | Причина |
|---|---:|---|
| Долг `>= 30000` | +4 | высокий долг |
| Долг `>= 10000` | +3 | значимый долг |
| Долг `>= 2000` | +2 | средний долг |
| Долг `> 0` | +1 | есть задолженность |
| Долг растет | +2 | ухудшение динамики |
| Нет оплат за 3 месяца | +3 | отсутствие платежной активности |
| Одна оплата за 3 месяца | +1 | слабая активность |
| Сумма оплат за 3 месяца `<= 0` | +2 | нет положительного погашения |
| Нет каналов контакта | +1 | труднее коммуницировать |
| Много мер за 3 месяца (`>= 5`) | +1 | уже была высокая интенсивность взыскания |
| Нет телефона и электронной квитанции | +1 | нет цифрового контакта |

Итоговый сегмент:

| Балл | Сегмент |
|---:|---|
| `>= 10` | `rule_critical` |
| `>= 7` | `rule_high` |
| `>= 4` | `rule_medium` |
| `< 4` | `rule_low` |

Почему rule-score нужен: бизнесу важно понимать, почему ЛС считается проблемным. Скоринг дает объяснимую основу: “долг высокий, оплат не было, контактных каналов нет”.

Пример: ЛС с долгом `40000`, ростом долга, отсутствием оплат и отсутствием контактов получит высокий балл и попадет в `rule_critical`.

### KMeans-кластеризация

Для поведенческой детализации используется KMeans по признакам:

```text
debt_start_t
charge_amt_t
paid_amt_t
debt_change_1m
payment_txn_count_l3m
payment_amount_l3m
action_total_l3m
action_unique_types_l3m
contact_channels_count
has_phone
has_e_receipt
can_remote_disconnect
```

Перед кластеризацией признаки стандартизируются:

```text
z = (x - mean) / std
```

Это нужно, потому что долг измеряется в рублях, а признаки контактов в `0/1`. Без стандартизации рублевые признаки полностью задавили бы остальные.

По умолчанию используется `MiniBatchKMeans` из sklearn, если он доступен. Если sklearn недоступен, включается встроенная реализация Lloyd KMeans на Python.

Почему MiniBatchKMeans: данных много, а MiniBatchKMeans быстрее классического KMeans на больших таблицах. Для бизнес-сегментации нам нужна не идеальная математическая оптимальность, а устойчивое разделение больших групп на интерпретируемые подтипы.

### Baseline-сегмент

Итог baseline-сегмента строится как комбинация rule-сегмента и ML-кластера:

```text
baseline_segment = rule_segment + "_c" + cluster_id
```

Пример:

```text
rule_high_c3
```

Это означает: ЛС находится в high-risk rule-группе, но внутри нее относится к поведенческому кластеру `3`.

### Hybrid business gates

Hybrid-логика сначала применяет жесткие бизнес-гейты, а затем запускает ML только там, где он допустим.

| Gate | Условие | ML? | Почему |
|---|---|---:|---|
| `gate_closed_no_debt` | `debt <= 0` | Нет | Нет долга, взыскание не требуется |
| `gate_soft_monitor` | долг `< 1000`, были платежи за 3 месяца | Нет | Мягкое наблюдение лучше агрессивной меры |
| `gate_prelegal_priority` | долг `>= 30000`, оплат нет | Да | Нужно приоритизировать внутри тяжелой группы |
| `gate_contact_deficit` | нет каналов контакта | Да | Нужно разделить сценарии поиска контакта |
| `gate_restriction_eligible` | возможно дистанционное отключение | Да | Нужно понять, кому ограничение потенциально эффективно |
| `gate_standard_collect` | все остальные | Да | Стандартный маршрут взыскания |

Внутри каждого ML-eligible gate дополнительно запускается KMeans. Это дает `hybrid_segment`, например:

```text
gate_standard_collect_c2
gate_prelegal_priority_c1
gate_soft_monitor__rule_only
```

Почему hybrid лучше только ML: модель не должна решать то, что уже запрещено или очевидно бизнес-правилами. Например, ЛС без долга не нужно отправлять в ML для выбора меры взыскания. А вот среди клиентов с высоким долгом ML полезен: он может отличить “совсем не платит и недоступен” от “платит редко, но реагирует на контакты”.

## Этап 5. Uplift-прототип: `build_uplift_prototype`

Uplift-этап отвечает на вопрос:

```text
Как изменится вероятность оплаты в следующем месяце, если применить конкретную меру?
```

По умолчанию оцениваются меры:

```text
sms
operator_call
restriction_notice
```

### Training / scoring split

Для выбранного месяца `score_month`:

| Часть данных | Как используется |
|---|---|
| Месяцы `< score_month` | обучение моделей |
| Месяц `score_month` | расчет рекомендаций |

Для каждой меры создается treatment-флаг:

```text
treatment = 1, если action_<measure>_l1m > 0
treatment = 0, иначе
```

Целевая переменная:

```text
paid_any_next_month
```

Пример: если для ЛС в январе была СМС, а в феврале была оплата, то для меры `sms` это обучающий пример `treatment = 1`, `outcome = 1`.

### Модель: T-learner

Для каждой меры обучаются две модели:

```text
model_treated:  P(payment next month | X, measure applied)
model_control:  P(payment next month | X, measure not applied)
```

Затем для каждого ЛС в score-месяце считается:

```text
p_if_treated = model_treated(X)
p_if_control = model_control(X)
uplift_score = p_if_treated - p_if_control
```

Если `uplift_score = 0.08`, это означает: модель ожидает прирост вероятности оплаты примерно на 8 процентных пунктов при применении меры.

### Алгоритм внутри модели

Используется `LogisticRegression` со `StandardScaler`.

Почему логистическая регрессия:

1. Она интерпретируема: можно смотреть коэффициенты и понимать, какие признаки влияют на вероятность оплаты.
2. Она хорошо подходит для бинарной цели `paid_any_next_month`.
3. Она быстрее и стабильнее на больших табличных данных, чем тяжелые модели.
4. Для прототипа важнее прозрачность и воспроизводимость, чем максимальная сложность.

Если в обучающих данных нет строк или есть только один класс, используется `ConstantProbModel`: модель возвращает постоянную вероятность. Это защитный механизм, чтобы редкие меры не ломали пайплайн.

### DR ATE

Для оценки среднего эффекта считается doubly robust estimate (`ate_dr`). Дополнительно обучается propensity-модель:

```text
P(treatment | X)
```

DR-оценка комбинирует:

1. прогноз outcome-моделей;
2. поправку на вероятность получить воздействие.

Почему это полезно: исторические меры назначались не случайно. Например, звонки оператора чаще могли получать более сложные должники. Простое сравнение “кому звонили” против “кому не звонили” будет смещено. DR-подход помогает снизить это смещение.

### Рекомендации

Для каждого ЛС выбирается мера с максимальным положительным uplift:

```text
recommended_measure = argmax(uplift_score)
```

Но есть ограничения:

| Условие | Что происходит | Причина |
|---|---|---|
| `gate_ml_eligible = 0` | мера не назначается | business-gate запрещает ML-рекомендацию |
| лучший uplift `<= 0` | мера не назначается | ожидаемый эффект не положительный |
| лучший uplift `> 0` | назначается лучшая мера | есть ожидаемый прирост оплаты |

Пример:

| ЛС | sms uplift | operator_call uplift | restriction_notice uplift | recommendation |
|---|---:|---:|---:|---|
| 101 | 0.02 | 0.09 | 0.03 | `operator_call` |
| 102 | -0.01 | 0.00 | -0.02 | no action |
| 103 | 0.10 | 0.12 | 0.05 | no action, если `gate_ml_eligible = 0` |

### Выходы uplift-этапа

| Файл | Что содержит |
|---|---|
| `uplift_measure_scores.csv` | uplift-score по каждому ЛС и мере |
| `uplift_recommendations.csv` | итоговая рекомендация по каждому ЛС |
| `uplift_measure_summary.csv` | сводка качества/эффекта по мерам |
| `uplift_feature_importance.csv` | признаки с максимальной разницей коэффициентов treated/control моделей |
| `load_metrics_uplift.json` | технические метрики запуска |

## Почему выбран именно такой набор решений

| Решение | Почему оно выбрано | Как помогает задаче |
|---|---|---|
| Raw-слой с `source_file/source_row_num` | Нужна трассируемость до исходной строки | Ошибки можно объяснить и проверить |
| Нормализованные dimension/fact таблицы | Табличная модель проще для аналитики и ML | Все источники соединяются по `ls_id` и датам |
| DQ как отдельная таблица | Входные данные шумные, но пайплайн не должен падать от каждой ошибки | Можно продолжать расчет и отдельно работать над качеством |
| Месячная витрина | Управление взысканием и лимиты мер обычно месячные | Признаки соответствуют реальному управленческому циклу |
| Leakage-safe target `T+1` | Модель не должна видеть будущее | Рекомендации ближе к реальному применению |
| Rule-score | Нужна объяснимость и бизнес-контроль | Понятно, почему ЛС рискованный |
| KMeans | Нужно найти поведенческие подтипы без ручной разметки | Внутри одного risk-сегмента появляются разные сценарии работы |
| Hybrid gates | Не все решения можно отдавать ML | Бизнес-правила защищают от недопустимых рекомендаций |
| T-learner uplift | Нужно сравнить эффект действия и бездействия для каждого ЛС | Рекомендации основаны на ожидаемом приросте, а не на частоте меры в истории |
| LogisticRegression | Нужна быстрая, интерпретируемая базовая модель | Результат можно объяснить и развивать дальше |
| DR ATE | Историческое назначение мер не случайно | Снижает смещение при оценке среднего эффекта |
| Web-интерфейс | Пайплайн должен быть наглядным и запускаться одной точкой | Пользователь видит ход обработки и результат без ручных команд |

## Ограничения текущего прототипа

1. Uplift-модель не является финальной causal inference системой: это рабочий прототип, который нужно валидировать A/B-тестом.
2. Лимиты мер пока загружаются как справочник, но не используются для глобальной оптимизации портфеля.
3. Стоимость мер не учитывается в функции выбора: выбирается максимальный положительный uplift, а не максимальная экономическая прибыль.
4. Исторические данные могут содержать смещение назначения мер; DR-оценка снижает риск, но не заменяет эксперимент.
5. KMeans дает удобные подтипы, но не гарантирует бизнес-идеальную сегментацию без последующей интерпретации профилей.

## Как развивать проект дальше

1. Добавить портфельный оптимизатор с учетом лимитов из `dim_action_limit` и стоимости каждой меры.
2. Подключить A/B-разметку для валидации uplift-рекомендаций.
3. Добавить мониторинг drift: изменение распределения долга, оплат, сегментов и uplift-score.
4. Расширить набор мер в uplift-модели.
5. Добавить business KPI: cost-to-collect, recovered amount, recovery speed.
6. Сформировать production-режим с регулярным запуском по расписанию.

## Короткий пример интерпретации результата

Допустим, ЛС имеет долг `35 000`, за последние 3 месяца не было оплат, каналов контакта мало, но возможно дистанционное отключение.

Пайплайн обработает его так:

1. В raw-слое сохранит исходные строки по профилю, балансу, оплатам и мерам.
2. В normalized-слое приведет ЛС, даты и суммы к единому виду.
3. В mart-слое сформирует строку за месяц `T`: долг, динамика, история оплат, история мер, признаки контакта.
4. Rule-score даст высокий риск из-за долга и отсутствия оплат.
5. Hybrid-gate может направить ЛС в `gate_restriction_eligible`.
6. KMeans внутри gate определит поведенческий подтип.
7. Uplift-модель сравнит ожидаемый эффект `sms`, `operator_call`, `restriction_notice`.
8. Если `restriction_notice` дает лучший положительный uplift, а gate допускает ML, в `uplift_recommendations.csv` появится рекомендация `restriction_notice`.
9. В презентационных графиках этот ЛС попадет в соответствующий gate, ML-eligible группу и распределение рекомендаций.

Такой путь дает не просто “модель сказала”, а проверяемую цепочку от исходной строки до объяснимой рекомендации.
